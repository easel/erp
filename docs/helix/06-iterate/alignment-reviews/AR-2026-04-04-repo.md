## Summary

The implementation is minimal -- scaffolding only -- and correctly reflects the WP-0 deliverables that have been completed. ## Methodology

All documents in the HELIX artifact stack were read in full: product vision, PRD, all 9 feature specs (FEAT-001 through FEAT-009), all 11 ADRs (ADR-001 through ADR-011), and all 3 solution designs (SD-001, SD-002, SD-003). All implementation source files in packages/shared/src, packages/server/src, and packages/web/src were read. Configuration files (package.json, tsconfig.json, docker-compose.yml, biome.json, CI workflow) were reviewed. Tracker state was queried via `ddx bead list` and `ddx bead ready`.

- The Vision references "Kratos quantumCMD / epoch IPS connector" as an integration priority; the PRD captures this as OAM-009 (P2), appropriately deferred.

**Finding:** No material misalignments between Vision and PRD.

### 4. ADRs <-> Solution Design Alignment

**ADR-002 (UUID v4) vs SD-001/SD-002:**
- SD-001 Section 3.3 correctly states "Entity identity via UUID v4 (per ADR-002)."
- SD-002 Section 2.1 states "All tables use UUID primary keys generated by `gen_random_uuid()`."
- ADR-002 notes SD-001 was updated from ULID to UUID. Alignment confirmed.

**ADR-003 (NUMERIC(19,6)) vs SD-001/SD-002:**
- SD-002 Section 1 states "Monetary amounts use NUMERIC(19,6); every amount is paired with a currency_code column."
- ADR-003 notes SD-001 was updated from BIGINT to NUMERIC(19,6). Alignment confirmed.

**ADR-004 (Audit Entry) vs SD-002:**
- SD-002 defines `audit_entry` table (not `audit_log`). Correct per ADR-004.
- ADR-004 instructs adding `chain_hash BYTEA` and `user_email VARCHAR(255)` to SD-002. The SD-002 table definition was only partially read, but the ADR states these columns should be added. This needs verification.

**ADR-006 (Compliance Status) vs SD-002:**
- ADR-006 mandates `compliance_status` on `sales_order`, `purchase_order`, `shipment`, and `quote`.
- SD-002 was noted in the ADR as needing this column added. The data model document was read partially; the column presence on all four tables should be verified when the full schema is implemented.

**ADR-007 (Four-State Period) vs SD-002:**
- SD-002 `fiscal_period.status` shows `CHECK (status IN ('FUTURE', 'OPEN', 'SOFT_CLOSED', 'HARD_CLOSED'))` -- correct per ADR-007.

**ADR-008 (Redis Optionality) vs SD-001:**
- SD-001 architecture diagram shows Redis as a data layer component. ADR-008 mandates Redis be labeled "optional enhancement" in diagrams and configs. The SD-001 diagram does not label Redis as optional.
- docker-compose.yml includes Redis without marking it as optional.

**ADR-009 (Bun + Local-First) vs SD-001:**
- SD-001 correctly references Bun runtime.
- SD-002 Section 2.4 adds sync metadata columns per ADR-009.

**SD-001 Monorepo Structure Mismatch (HIGH):**
- SD-001 Section 4 specifies an `apps/api/` + `apps/web/` + `packages/` structure with pnpm workspaces and Turborepo.
- Actual implementation uses `packages/shared/`, `packages/server/`, `packages/web/` with Bun workspaces (not pnpm, no Turborepo).
- SD-001 specifies separate `packages/db/`, `packages/platform/`, `packages/finance/`, `packages/sales/`, etc.
- The implementation collapses the API server into `packages/server/` directly.
- SD-001 references `pnpm-workspace.yaml` and `turbo.json`; neither exists. `package.json` uses `"workspaces": ["packages/*"]` (Bun-native workspaces).

**SD-001 Testing Framework Mismatch (MEDIUM):**
- SD-001 Section 2 specifies Vitest for unit/integration tests.
- Implementation uses `bun test` (Bun's built-in test runner). Test files import from `bun:test`.
- SD-003 WP-0 also references "Vitest" but the actual scaffold uses `bun test`.

**MoneyAmountSchema Defect (existing finding):**
The regex `/^\d+(\.\d{1,6})?$/` allows unlimited integer digits, but ADR-003 specifies NUMERIC(19,6) which limits to 13 integer digits. This is already tracked as hx-97b53176.

**Server Entrypoint Missing (existing finding):**
`packages/server/src/index.ts` exports `buildApp` but does not actually start a server. The `dev` and `start` scripts reference `src/index.ts` which would only export, not listen. Tracked as hx-9164fdf4.

**Missing MoneyAmountSchema negative support:**
ADR-011 states MoneyInput "supports negative amounts (credits, adjustments)" but `MoneyAmountSchema` regex requires non-negative amounts (`^\d+`). This may be intentional at the schema level (store absolute value with debit/credit indicator) but should be explicitly documented.

## Recommended Actions

### Priority 1 (Before WP-1 starts)

1. **Update SD-001 monorepo layout** (AR-001): Replace the pnpm/Turborepo/apps+packages structure with the actual Bun workspace flat `packages/` layout. This is the most consequential documentation drift -- a new contributor reading SD-001 would set up the wrong structure.

2. **Update SD-001 and SD-003 testing references** (AR-002): Replace Vitest references with `bun test`. Document the rationale: Bun's built-in test runner eliminates a dependency and aligns with ADR-009's preference for Bun built-in tools.

3. **Resolve WP-0 remaining items** (AR-008): Decide whether auth scaffolding (Keycloak dev instance) and observability (structured logging, OpenTelemetry) belong in WP-0 or WP-1. Update SD-003 accordingly and create beads if WP-0.

### Priority 2 (During WP-1)

4. **Fix MoneyAmountSchema** (AR-004/hx-97b53176): Enforce 13-digit integer limit per NUMERIC(19,6).

5. **Add Redis optionality labels** (AR-003): Annotate SD-001 diagram and docker-compose.yml per ADR-008.

6. **Resolve negative money question** (AR-005): Document in ADR-003 addendum whether `Money.amount` can be negative.

7. **Add server entrypoint** (AR-007/hx-9164fdf4): Server must be startable.

### Priority 3 (Before WP-2)

8. **Update SD-001 ESLint reference** (AR-012): Replace eslint-plugin-boundaries with Biome-compatible strategy.

9. **Add Mailpit to docker-compose or update SD-003** (AR-006).
