# Data Model: SatERP — Phase 1

**Authority Level:** 4 (Design)
**Status:** Draft
**Created:** 2026-04-04
**Governed by:** [PRD](../../01-frame/prd.md)

> **Scope:** This document covers Phase 1 (P0) entities only. Phase 2 entities — including orbital asset management, program management, and advanced revenue recognition — will be designed in a separate document.

---

## 1. Design Principles

| Principle | Implementation |
|---|---|
| Multi-entity by default | Every transactional table carries `entity_id` FK to `legal_entity` |
| Immutable audit | Soft deletes via `deleted_at`; `version` column for optimistic locking; `audit_entry` table is append-only |
| Currency precision | Monetary amounts use `NUMERIC(19,6)`; every amount is paired with a `currency_code` column |
| Temporal awareness | Effective/expiry dates on rate tables; `fiscal_period` drives GL posting; all timestamps are `TIMESTAMPTZ` |
| ITAR compartmentalization | Row-level security policies keyed on `itar_compartment_id`; users are granted per-compartment access |
| Referential integrity | All foreign keys enforced at the database level; no application-only constraints |
| Idempotent operations | Natural keys / unique constraints on external references to support safe retries |

## 2. Common Patterns

### 2.1 Primary Key

All tables use `UUID` primary keys generated by `gen_random_uuid()`.

### 2.2 Audit Columns

Every mutable table includes:

| Column | Type | Constraint | Notes |
|---|---|---|---|
| `created_at` | `TIMESTAMPTZ` | `NOT NULL DEFAULT now()` | Immutable after insert |
| `created_by` | `UUID` | `NOT NULL` FK to `user_account` | |
| `updated_at` | `TIMESTAMPTZ` | `NOT NULL DEFAULT now()` | Updated on every write |
| `updated_by` | `UUID` | `NOT NULL` FK to `user_account` | |
| `version` | `INTEGER` | `NOT NULL DEFAULT 1` | Optimistic concurrency; increment on update |
| `deleted_at` | `TIMESTAMPTZ` | `NULL` | Non-null means soft-deleted |

### 2.3 Multi-Entity Tenancy

Transactional tables include:

| Column | Type | Constraint |
|---|---|---|
| `entity_id` | `UUID` | `NOT NULL` FK to `legal_entity` |

Row-level security policies enforce entity isolation where required.

### 2.4 Sync Metadata (Local-First, per ADR-009)

Offline-capable tables include additional sync columns:

| Column | Type | Description |
|--------|------|-------------|
| sync_version | BIGINT NOT NULL DEFAULT 0 | Monotonically increasing version, incremented on each mutation. Used by sync protocol for change detection. |
| last_synced_at | TIMESTAMPTZ | Timestamp of last successful sync to/from server. NULL for records created server-side that have not been pulled by any client. |
| sync_status | VARCHAR(20) DEFAULT 'synced' | CHECK (sync_status IN ('synced', 'pending_push', 'conflict')). Status of this record's sync state. |

**Offline tier assignments:**
- **Tier 1 (always offline):** customer, vendor, product, crm_company, crm_contact, inventory_item, inventory_level, screening_list_entry, country_restriction, restricted_region
- **Tier 2 (eventual sync):** quote, sales_order, purchase_order, journal_entry, customer_invoice, vendor_bill, activity, opportunity, lead
- **Tier 3 (online-only, no sync columns):** audit_entry, payment, payment_batch, shipment (compliance_status requires live verification)

### 2.5 Enum Strategy

Small, stable value sets use PostgreSQL `ENUM` types. Larger or user-extensible sets use reference tables with a `code` / `label` pattern.

### 2.6 JSONB Extension Fields

Tables that need user-extensible metadata include an `ext` column of type `JSONB DEFAULT '{}'::jsonb`.

| Column | Type | Constraint | Notes |
|---|---|---|---|
| `id` | `UUID` | `PK DEFAULT gen_random_uuid()` | |
| `user_id` | `UUID` | `NOT NULL` FK `user_account(id)` | |
| `role_id` | `UUID` | `NOT NULL` FK `role(id)` | |
| `entity_id` | `UUID` | FK `legal_entity(id)` | NULL = global assignment |
| `program_id` | `UUID` | FK (program table, Phase 2) | NULL = not program-scoped |
| `compartment_id` | `UUID` | FK `itar_compartment(id)` | NULL = not compartment-scoped |
| `effective_from` | `TIMESTAMPTZ` | `NOT NULL DEFAULT now()` | When this assignment becomes active |
| `effective_until` | `TIMESTAMPTZ` | | NULL = no expiration |
| `granted_by` | `UUID` | `NOT NULL` FK `user_account(id)` | |
| `revoked_at` | `TIMESTAMPTZ` | | Non-null = revoked |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL DEFAULT now()` | |
| `created_by` | `UUID` | `NOT NULL` FK `user_account(id)` | |

| `updated_by` | `UUID` | `NOT NULL` FK `user_account(id)` | |

This table enables four scoping patterns:
- **Entity-scoped:** `entity_id` set, others NULL
- **Program-scoped:** `entity_id` + `program_id` set
- **ITAR-compartment-scoped:** `entity_id` + `compartment_id` set
- **Time-bounded:** any of the above with `effective_from` / `effective_until` set

## 11. Partitioning Strategy

### 11.1 Partitioned Tables

| Table | Partition Key | Strategy | Retention |
|---|---|---|---|
| `audit_entry` | `occurred_at` | Range, monthly | 7 years (regulatory) |
| `journal_entry_line` | (via `journal_entry.fiscal_period_id`) | Range, by fiscal period | Indefinite |
| `screening_result` | `screening_date` | Range, monthly | 7 years |
| `tracking_event` | `event_timestamp` | Range, monthly | 3 years |
| `notification_delivery` | `attempted_at` | Range, monthly | 1 year |

### 11.2 Partition Management

- New partitions are created automatically via `pg_partman` or a scheduled function at least 3 months ahead.
- Old partitions beyond retention are detached and archived to cold storage (S3/GCS via `pg_dump`) before dropping.
- Partition boundaries align to calendar months at UTC midnight.

## 12. Materialized Views

### `mv_gl_balance_summary`

Aggregated GL balances across all segments for dashboard reporting.

```sql
SELECT entity_id, account_id, fiscal_period_id, currency_code,
       SUM(period_debit_total) AS period_debit,
       SUM(period_credit_total) AS period_credit,
       SUM(period_net) AS period_net,
       SUM(ytd_net) AS ytd_net
FROM gl_balance
GROUP BY entity_id, account_id, fiscal_period_id, currency_code;
```

**Refresh:** On period close and on demand.

### `mv_ar_aging`

Accounts receivable aging buckets (current, 1-30, 31-60, 61-90, 90+).

```sql
SELECT entity_id, customer_id, currency_code,
       SUM(CASE WHEN due_date >= CURRENT_DATE THEN balance_due ELSE 0 END) AS current_amount,
       SUM(CASE WHEN due_date BETWEEN CURRENT_DATE - 30 AND CURRENT_DATE - 1 THEN balance_due ELSE 0 END) AS days_1_30,
       SUM(CASE WHEN due_date BETWEEN CURRENT_DATE - 60 AND CURRENT_DATE - 31 THEN balance_due ELSE 0 END) AS days_31_60,
       SUM(CASE WHEN due_date BETWEEN CURRENT_DATE - 90 AND CURRENT_DATE - 61 THEN balance_due ELSE 0 END) AS days_61_90,
       SUM(CASE WHEN due_date < CURRENT_DATE - 90 THEN balance_due ELSE 0 END) AS days_over_90,
       SUM(balance_due) AS total_outstanding
FROM customer_invoice
WHERE status NOT IN ('PAID', 'VOID', 'WRITTEN_OFF') AND deleted_at IS NULL
GROUP BY entity_id, customer_id, currency_code;
```

**Refresh:** Daily, and on demand.

### `mv_ap_aging`

Accounts payable aging buckets (same structure as AR aging).

```sql
SELECT entity_id, vendor_id, currency_code,
       SUM(CASE WHEN due_date >= CURRENT_DATE THEN balance_due ELSE 0 END) AS current_amount,
       SUM(CASE WHEN due_date BETWEEN CURRENT_DATE - 30 AND CURRENT_DATE - 1 THEN balance_due ELSE 0 END) AS days_1_30,
       SUM(CASE WHEN due_date BETWEEN CURRENT_DATE - 60 AND CURRENT_DATE - 31 THEN balance_due ELSE 0 END) AS days_31_60,
       SUM(CASE WHEN due_date BETWEEN CURRENT_DATE - 90 AND CURRENT_DATE - 61 THEN balance_due ELSE 0 END) AS days_61_90,
       SUM(CASE WHEN due_date < CURRENT_DATE - 90 THEN balance_due ELSE 0 END) AS days_over_90,
       SUM(balance_due) AS total_outstanding
FROM vendor_bill
WHERE status NOT IN ('PAID', 'VOID') AND deleted_at IS NULL
GROUP BY entity_id, vendor_id, currency_code;
```

**Refresh:** Daily, and on demand.

### `mv_inventory_valuation`

Inventory value by item, location, and lot.

```sql
SELECT il.entity_id, il.inventory_item_id, ii.item_code, ii.name,
       il.location_id, il.lot_id,
       il.quantity_on_hand,
       ii.standard_cost,
       ii.cost_currency_code AS currency_code,
       (il.quantity_on_hand * COALESCE(ii.standard_cost, 0)) AS total_value
FROM inventory_level il
JOIN inventory_item ii ON ii.id = il.inventory_item_id
WHERE il.quantity_on_hand > 0;
```

**Refresh:** On inventory transaction and nightly.

### `mv_pipeline_summary`

Sales pipeline summary by stage for forecasting.

```sql
SELECT o.entity_id, ps.code AS stage_code, ps.name AS stage_name,
       COUNT(*) AS opportunity_count,
       SUM(o.amount) AS total_amount,
       o.currency_code,
       SUM(o.amount * COALESCE(o.probability, ps.win_probability, 0) / 100) AS weighted_amount
FROM opportunity o
JOIN pipeline_stage ps ON ps.id = o.pipeline_stage_id
WHERE o.deleted_at IS NULL
  AND NOT ps.is_closed_won AND NOT ps.is_closed_lost
GROUP BY o.entity_id, ps.code, ps.name, o.currency_code;
```

**Refresh:** Hourly, and on demand.
